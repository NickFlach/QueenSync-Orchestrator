Update QueenSync v0.1 into QueenSync v0.2 with external Kannaka site integration and universal agent onboarding.

Deployment target:
- Production domain: ninja-portal.com
- Existing active Kannaka interfaces:
  - radio.ninja-portal.com
  - observatory.ninja-portal.com

New core requirement:
QueenSync must support onboarding ANY agent, not just built-in Kannaktopus arms.

Add “Agent Onboarding” as a first-class feature.

Agent types:
- human-configured agent
- API agent
- local simulated agent
- Replit-hosted agent
- OpenClaw agent
- external webhook agent
- future MCP-compatible agent

Agent onboarding flow:
1. User clicks “Onboard Agent”
2. User enters:
   - agent name
   - agent type
   - endpoint URL
   - capabilities
   - auth method: none, API key, bearer token, future JWT
   - heartbeat URL optional
   - description / role
3. QueenSync tests connection if endpoint provided
4. Agent is added to registry
5. Agent becomes eligible for task routing
6. Agent actions are logged
7. Agent outputs pass through Memory Gate

Agent schema:
{
  "agent_id": "uuid",
  "name": "Kannaka Radio Agent",
  "type": "external_webhook",
  "endpoint_url": "https://radio.ninja-portal.com/api/agent",
  "heartbeat_url": "https://radio.ninja-portal.com/api/health",
  "capabilities": ["radio", "signal_ingestion", "transmission_analysis"],
  "auth_method": "none",
  "status": "pending",
  "description": "Radio signal interface for Kannaka transmissions"
}
